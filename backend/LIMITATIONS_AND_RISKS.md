## 2. Evidence Threshold

### Текущее поведение
- `EVIDENCE_MIN_SCORE = 0.30` (default из `config.py`)
- Единый источник truth: `rag/evidence_builder.py` читает `EVIDENCE_MIN_SCORE` из env
- `rag/classifier.py` импортирует `MIN_EVIDENCE_SCORE` из `evidence_builder` — не дублирует
- `/classify/explain` возвращает `evidence_threshold_used` и `evidence_threshold_source`

### Риск
- Порог `0.30` **не откалиброван** на реальных данных ТН ВЭД
- При малой базе знаний (мало Excel-записей или PDF) пороговое значение будет недостигаемым
- При `EVIDENCE_MIN_SCORE=0.1` (dev) система пропускает классификации с низкой доказательной базой

### Правило
**Production ЗАПРЕЩЕНО возвращать код если `evidence.evidence_score < threshold`.**
Нарушение = потенциальная ошибка классификации ценой миллионов рублей.

### Рекомендация
Откалибровать порог на выборке из 100–200 реальных товаров с известными кодами ТН ВЭД.

## 3. PDF и text_quality_score

### Проблема
PDF-пояснения ЕЭК могут содержать текст, извлечённый с ошибками:
- Сканированные страницы без распознавания (garbage text)
- Кириллица в нестандартных кодировках
- Специальные символы (·□▪▸►▶) вместо нормального текста

### Решение
`ingestion/pdf_extractor.py` вычисляет `text_quality_score` для каждого чанка:
- `1.0` — текст полностью читаем
- `< 0.40` — чанк помечается как низкого качества (`text_quality_warning`)
- Чанки с `score < 0.40` продолжают индексироваться, но видны в `/classify/explain`

### Где видно
`/classify/explain` → `evidence.pdf_chunks[].text_quality_score`
`/classify/explain` → `evidence_warnings` — предупреждение если `pdf_chunks_found=0`

## 4. LLM (qwen2.5:7b) — ограничения

- **Галлюцинации кодов:** LLM может предложить несуществующий код ТН ВЭД
  → Mitigation: Evidence First (блокировка без доказательств)
- **Неправильное применение ОПИ:** LLM не читает нормативные документы
  → Mitigation: Rule Engine применяет ОПИ программно
- **Языковой дрейф:** модель смешивает русский и другие языки в рассуждениях
  → Mitigation: структурированный промпт с жёсткими инструкциями
- **Скорость:** ~30–60 сек на запрос без GPU

## 5. Эвристики (не нормативные методы)

Следующие компоненты содержат эвристики — они НЕ являются официальными правилами ТН ВЭД:

| Компонент | Эвристика |
|---|---|
| `evidence_builder.py` | Веса доказательств: Excel=0.40, PDF=0.30, Notes=0.15, Rank=0.15 — эмпирические |
| `validator.py` | `_check_exclusions` — ключевые слова для поиска исключений |
| `validator.py` | `_find_competing_codes` — алгоритм поиска конкурирующих кодов |
| `rule_engine.py` | Классификация по ОПИ — алгоритмическое приближение, не юридическое толкование |
| `devil_advocate.py` | Adversarial check — LLM-оценка, может ошибаться |
| `retriever.py` | `CHAPTER_BOOST_FACTOR=1.3` — коэффициент усиления главы ТН ВЭД |

**Вывод:** система — инструмент предварительной классификации. Окончательное решение принимает квалифицированный таможенный специалист.

- Ollama запускается локально

## 7. Строго запрещено

1. Использовать для окончательной таможенной декларации без проверки специалиста
2. Снижать `EVIDENCE_MIN_SCORE` ниже `0.2` в production
3. Игнорировать `requires_clarification=true` и возвращать код пользователю
4. Возвращать `code != null` при `evidence.is_sufficient=false`
5. Применять без реального Excel ТН ВЭД (полная база, не тестовая)

---

## 8. Требования для production

- [ ] Ollama запущен, модель загружена
- [ ] Excel ТН ВЭД (полный, >13 000 записей) в `data/excel/`
- [ ] PDF-пояснения ЕЭК в `docs/explanations/` (опционально, но рекомендуется)
- [ ] `build_knowledge_base.py` выполнен с `PDF_DIRS=./data/pdf,./docs/explanations`
- [ ] `/health` показывает `codes_count > 1000`, `docs_explanations_included=true`
- [ ] Real-mode тест пройден на 20+ реальных товарах с известными кодами
- [ ] `EVIDENCE_MIN_SCORE` не переопределён через env в production

## 9. Benchmark

Шаблон: `tests/real_cases_template.xlsx`

```bash
# Запуск benchmark на реальных кейсах:
python tests/benchmark.py --cases tests/real_cases_template.xlsx --output results/
```

Целевые метрики (на реальных данных с полной базой):
- Точное совпадение кода (10 цифр): >70%
- Совпадение группы (4 цифры): >85%
- Отказ классифицировать при неопределённости: >95% (precision важнее recall)
