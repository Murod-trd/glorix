**Marketplace.jsx**: добавлен компонент `ProductImage` (после импортов, до `Stars`):
```jsx
function ProductImage({ product, style = {} }) {
  const [imgFailed, setImgFailed] = useState(false);
  if (product.photo && !imgFailed) {
    return (
      <img src={product.photo} alt={product.title}
        onError={() => setImgFailed(true)}
        style={{ width: '100%', height: '100%', objectFit: 'cover', display: 'block', ...style }}
      />
    );
  }
  return <ProductIllustration id={product.photoId} />;
}
```
Заменены 3 вхождения `<ProductIllustration id={product.photoId} />`:
- ProductModal (блок 260px × полная ширина)
- ProductCard (блок 180px)
- CartPanel (миниатюра 50px)

Пикер SVG-иллюстраций в AddProductModal оставлен без изменений (там правильно показывать SVG при выборе типа).

**Архитектура fallback**: product.photo → `<img onError={...}>` → SVG. Если Unsplash недоступен в CIS, пользователь видит SVG без каких-либо ошибок.

**Задача 2 — КП из AddProductModal: нет ТН ВЭД, нет таблицы:**

### Модель прав ролей разделена: покупка в маркетплейсе ≠ создание тендера; тендеры реально публикуются и видны всем, покупатель анонимен до завершения

Основатель уточнил модель прав (полное разделение ролей по правам, «оба»
= объединение, решение принято после прямого запроса совета «как бизнес-

### КРИТИЧНО — «Мои товары» показывал весь каталог чужих товаров, не только товары текущего продавца

Основатель прислал скриншот: аккаунт продавца KazSteel Trading, раздел
называется «Мои товары» (32 найдено), но показывает подсолнечное масло,
шёлковую ткань, цемент — товары совершенно других продавцов.

**Корневая причина — давняя, не из этой сессии.** Заголовок «Мои товары»
для режима «только продавец» существовал в коде давно (`canSell &&
!canBuy ? 'Мои товары' : ...`), но сама страница ВСЕГДА показывала общий
каталог без какой-либо фильтрации по продавцу. Раньше (каталог из 6
товаров, без реального добавления) это было незаметно; теперь, когда
каталог расширен до 32 товаров и реально работает добавление товаров
(сессия 21), разница между заголовком и содержимым стала явной и
немедленно заметной основателю.

**Исправление.** Добавлена реальная фильтрация: для режима «только
продавец» страница показывает товары, где `seller.id` совпадает с id
текущего аккаунта (`currentUser.id`) — для покупателя и аккаунта «оба»
поведение не изменилось, весь общий каталог виден как и должно быть на
маркетплейсе.

**Попутно найдена и исправлена связанная проблема.** Демо-аккаунты
продавцов (`mock.js`: KazSteel Trading id `u2`, Sibmetall Group id `u3`)
никогда не были связаны с реальными товарами в каталоге — у товаров
свои отдельные id продавцов (`s1`-`s32`), не пересекающиеся с id
демо-аккаунтов. После фикса фильтрации это означало бы, что «Мои
товары» для ЛЮБОГО демо-аккаунта продавца оказались бы пустыми —
лучше, чем показывать чужие товары, но не идеально для демонстрации.
Связаны явно: товар «Алюминиевый профиль строительный» переоформлен на
продавца `u2` (KazSteel Trading), товары «Арматура А500С» и «Лист
стальной горячекатаный» — на `u3` (Sibmetall Group). Заодно унифицировано
название в `Analytics.jsx` (там оставалось старое «Sibmetall Trading»,
не синхронизированное с переименованием в предыдущей сессии).

### Что сделано в этой сессии — маркетплейс стал реально работающим внутри demo-сессии браузера

Основатель дал явный список из 8 требований: превратить витрину
маркетплейса в реально функционирующий внутри demo маркетплейс — продавец
размещает товар с рисунком и примечаниями, покупатель покупает один или
несколько товаров, есть корзина, санкционные товары запрещены, продажа
определённых категорий в Россию заблокирована по экспортным
ограничениям.

**Архитектурное решение, зафиксированное заранее.** Backend не
существует (см. критичный пробел №1 из анализа сессии 20) — поэтому
«реальная работа» означает: данные сохраняются в `localStorage` браузера
текущей вкладки. Товар, добавленный продавцом, реально появляется в
общем каталоге; покупка реально списывает остаток; корзина реально
работает — но всё это видно только в текущей browser-сессии, не
синхронизируется между устройствами/пользователями. Это честная,
осознанная граница demo-фазы, не обман.

**Новые модули:**
- `src/data/marketplaceStore.js` (новый) — централизованное хранилище:
  объединяет статичный демо-каталог (32 товара) с товарами, добавленными
  пользователем; отдельно хранит override остатка для статичных товаров
  (сам файл `marketplace.js` как код не модифицируется — это данные,
  не код); корзина; история заказов. Всё через localStorage с мягкой
  обработкой ошибок (переполнение/недоступность не ломает приложение).
- `src/context/CartContext.jsx` (новый) — реактивная корзина по тому же
  паттерну, что `AccountContext.jsx`.

**Реализация по пунктам запроса основателя:**

1-6. **Раздел стал настоящим маркетплейсом.** `Marketplace()` теперь
читает товары через `getAllProducts()` (объединённый список) с функцией
`refreshProducts()`, вызываемой после покупки/размещения товара —
каталог обновляется без перезагрузки страницы.

2. **Продавец размещает товар с рисунком.** Форма «Разместить товар»
теперь реально сохраняет товар (`addUserProduct()`) вместо простого
переключения шага без сохранения — это была корневая причина, почему
маркетплейс был «не настоящим». Добавлен выбор иллюстрации товара из
существующего набора `ProductIllustration.jsx` (без backend нет загрузки
настоящих фото — честное решение, не выдающее себя за реальное фото).
Список иллюстраций экспортирован как `PRODUCT_ILLUSTRATION_IDS`, чтобы
не дублировать его отдельно.

3. **Примечания от продавца.** Найден и закрыт реальный, отдельный от
основного запроса пробел: поле `description` уже проверялось санкционным
фильтром, но в форме физически не было input для его заполнения —
продавец не мог его ввести вообще. Добавлено текстовое поле «Примечания
продавца» с явным пояснением цели (отраслевой сленг, нюансы партии,
всё, что не укладывается в структурированные характеристики).

4-5. **Покупка одного или нескольких товаров + корзина.** В `ProductModal`
кнопка «Купить» разделена на «+ В корзину» и «Купить сейчас» (раздельные
сценарии). Создан `CartModal` — полноценное окно корзины с возможностью
оформить заказ на несколько разных товаров одним действием, отдельной
проверкой остатка и экспортных ограничений для каждой позиции, и общим
итогом на Escrow. Иконка корзины с счётчиком добавлена в шапку
маркетплейса (видна только покупателям).

7. **Санкционные товары запрещены** — этот механизм уже существовал
(`screenForSanctions`, закрыт в одной из предыдущих сессий), подтверждён
работающим повторным тестом в этой сессии (оружие блокируется).

8. **Экспортные ограничения по стране покупателя — новый механизм.**
Добавлена `checkExportRestriction()` в `sanctionsScreening.js`. Основание
— реальные, проверенные через поиск действующие санкционные пакеты ЕС
против России (20-й пакет, апрель 2026: расширенные экспортные запреты на
металлы, минералы, химикаты, промышленные инструменты и станки; Annex I
по товарам двойного назначения покрывает электронику). Категориальное
приближение (не точная классификация по ТН ВЭД/ECCN — честно отмечено в
коде): покупатель из России не может купить товары категорий «металлы»,
«химикаты», «электроника», «оборудование» — ни в `ProductModal`, ни в
`CartModal`, с явным объяснением причины в интерфейсе.

**Найдена и исправлена критическая ошибка в процессе работы**: форма
размещения товара сохраняла категорию как русский текст-надпись
(«Текстиль»), а не как id («textile»), используемый везде в системе для
фильтрации и проверок — это сломало бы и фильтр по категориям, и новую
проверку экспортных ограничений для любого нового товара. Исправлено до
коммита, не после.

**Протестировано:** полный набор unit-тестов бизнес-логики (8 сценариев
— добавление товара, списание остатка для пользовательских и статичных
товаров, корзина, заказы, экспортные ограничения по 4 комбинациям
категория/страна, санкционный скрининг оружия) — все прошли. React-рендер
для всех трёх типов аккаунта (покупатель/продавец/оба) — корректная
видимость кнопок корзины/размещения товара для каждой роли, без ошибок
рендера.

Проверено сборкой: `npm run build` проходит, главный чанк 733.39KB (рост

### Найден и закрыт системный класс багов — русские словоформы и ложные совпадения слов

Основатель показал скриншот: запрос «оборудования» дал случайные
результаты (лошади, ослы, свиньи) с переводом «of» вместо реального
слова.

**Корневая причина.** Словарь содержит «оборудование» (без окончания
-я), но запрос был «оборудовани**я**» — последняя буква меняется при
падежном склонении, а не просто добавляется (в отличие от «сталь» →
«стальная», где «сталь» — точный префикс). Прежняя защита от ложных
совпадений требовала точного префикса, поэтому словарь не находил
совпадение, поиск проваливался в живой перевод, который для этого
конкретного запроса вернул мусорный результат «of» (присущая
ненадёжность неофициального эндпоинта, см. Decision 10) — отсюда
случайные результаты.

**Исправление.** Добавлен стемминг: для русских словарных ключей длиной
от 4 символов перед сравнением отрезается последняя буква — это
покрывает абсолютное большинство падежных/числовых окончаний русского
языка одним простым правилом («оборудование» теперь совпадает с
«оборудования», «оборудованию» и подобными формами).

**Проактивный повторный скан после изменения правила.** Стемминг —
более широкое правило, чем точный префикс, поэтому он мог создать НОВЫЕ
ложные совпадения, которых не было раньше. Написан скрипт, проверивший
весь словарь (378 ключей) на потенциальные коллизии с учётом нового
стемминга. Найдено 9 дополнительных реальных коллизий (не были
обнаружены раньше, потому что точный префикс их не давал): стекло
vs стекловолокно/стеклотара/стекловата, картон vs картофель, плата vs
платина, вода vs водород, сера vs серебро, хлор vs хлопок, обои vs
оборудование, соль vs солома, медь vs медикаменты. Все добавлены в
`RUSSIAN_COLLISION_KEYS` с пояснением для каждой пары.

**Побочная находка при тестировании — ложное совпадение на английской
стороне.** «серебро» (→ silver) совпадало внутри названия рыбы «...silver
pomfrets» (серебристый помфрет) — это не ошибка стемминга, а присущее
ограничение точного слова в датасете: некоторые виды рыб/животных в
международной номенклатуре называются с использованием слов металлов
как часть видового имени. Решено НЕ через общее правило сортировки
(первая попытка — приоритет совпадениям в начале строки — сломала «газ»,
который стал показывать «Gas-tight cabinets» вместо «Petroleum gases»;
откатано), а через узкий явный список конкретных кодов-исключений
(`MATERIAL_WORD_FALSE_POSITIVE_CODES`), найденных целевым сканированием
пересечения металлов/материалов с рыбными/животными терминами в
датасете.

**Урок, зафиксированный явно:** универсальные эвристики сортировки по
позиции слова в строке рискованны — то, что решает одну проблему
(серебро/рыба), может создать другую (газ/газонепроницаемый). Узкие,
явные списки конкретных найденных случаев — менее элегантное, но более
предсказуемое и безопасное решение для этого типа задачи.

Проверено: полный регрессионный тест из 28 запросов (оборудования,
оборудование, медная труба, труба пвх, стальная труба, насос, насосы,
олово, оружие, вино, мел, газ, лук, чай, мёд, медь, стекло,
стекловолокно, картон, картофель, плата, платина, вода, водород, сера,
серебро, хлор, хлопок, обои) — все дают релевантный результат, без
регрессий по сравнению с предыдущими сессиями.

ПВХ. Дополнительно поставлен явный ультиматум: если задача невыполнима —
сказать прямо заранее, не сжигать токены на недоведённую работу.

**Реальная картина — баг имел три отдельные причины, исправленные в
этой сессии одна за другой через прямое тестирование каждой:**

1. **OR вместо AND между несколькими совпавшими словарными терминами.**
   Когда совпадало два термина («труба» и «пвх»), система искала товары,
   подходящие под ЛЮБОЙ из двух переводов, а не под оба сразу — это и
   давало любую трубу (сталь, медь, алюминий) без связи с ПВХ. Добавлена
   `matchAllPhrases()` — требует совпадения всех фраз одновременно, с
   мягким откатом на OR только если AND даёт пустой результат.

2. **Правило "короткие слова требуют полную границу" сломало
   множественное число обычных терминов.** Слово «tube» (4 символа)
   требовало полного совпадения `\btube\b`, но в датасете везде «tubes»
   (множественное число) — поэтому даже правильный товар (труба ПВХ,
   код 391723, реально содержит и «tube», и «vinyl chloride») не находился
   AND-логикой. Исправлено: вместо общего правила по длине слова — точный
   список реально опасных коротких слов (`tin`, `car`, `gas`), которым
   нужна полная граница; остальные слова получили разрешённое окончание
   `e?s?` для множественного числа.

3. **Та же категория коллизии — но на стороне русских ключей словаря.**
   Ключ «мед» (→ honey) совпадал как подстрока внутри «медная» (медь,
   металл), показывая мёд в результатах поиска медных труб. Добавлена
   `matchesRussianTerm()` с явным списком русских коллизионных ключей
   (`RUSSIAN_COLLISION_KEYS`: мед/мёд, вино, мел, газ, лук, чай — найдены
   проактивным сканированием всего словаря на похожие случаи, а не
   дожиданием следующего скриншота с такой же ошибкой).

**Проактивное сканирование (выполнено по своей инициативе, не дожидаясь
следующей жалобы).** Написан скрипт, который проверил все 378 однословных
ключей словаря на потенциальные коллизии (короткий ключ — префикс другого
слова). Найдено и исправлено: «мел» совпадал внутри официального названия
товарной группы «редкозе**мел**ьных металлов» (отдельная, четвёртая причина
бага — обнаружена только благодаря этому сканированию, не входила в
исходную жалобу основателя). Добавлена `matchesWordInText()` — отдельная
функция с безусловной двусторонней границей слова для проверки
произвольного запроса пользователя против названий товарных групп
(словарные ключи и запросы пользователя — это разные категории текста с
разными требованиями к границе).

**Попутное улучшение.** Товары внутри одной официальной товарной группы
сортировались по числовому коду, а не по релевантности — поэтому «чай»
показывал кофе первым (группа 09 объединяет кофе и чай официально).
Добавлена сортировка по совпадению со словарным переводом, если он есть —
«чай» теперь показывает чай первым.

**Подтверждено прямым тестированием 13 запросов** (медная труба, труба
пвх, стальная труба, насос, олово, оружие, вино, мел, газ, лук, чай, мёд,
медь) — все дают релевантный товар на первой позиции, без единой
регрессии по сравнению с предыдущими сессиями.

**Что я не успел и явно не утверждаю как сделанное:** проактивное
сканирование покрыло только однословные ключи на коллизию с другими

не сработали. Протестировано тремя сценариями: Google работает (MyMemory
не вызывается), Google отказывает и MyMemory подхватывает, оба отказывают
(мягкий возврат без ошибки) — все три подтверждены логически корректными.
