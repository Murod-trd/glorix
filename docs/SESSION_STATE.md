Это единственный файл, который заменяет историю чата. Любой новый чат должен
прочитать его первым и продолжать работу, как если бы чат не менялся —
без повторных вопросов "с чего начать" и без пересказа уже известного.

**Правило обновления**: при каждом значимом изменении (исправлен баг,
принято решение, закрыт пункт аудита) — обновляется секция "ТЕКУЩАЯ ТОЧКА"

низу файла с датой. Это даёт одновременно: (1) актуальный срез сейчас,
(2) полную историю того, как мы сюда пришли — без необходимости читать чаты.

### Что сделано в этой сессии — поиск ТН ВЭД теперь работает по любому товару

Основатель сообщил: поиск кодов ТН ВЭД в Document Center находил только 6
товаров из захардкоженного списка (пшеница, цемент, арматура, полиэтилен,
масло подсолнечное + одна позиция текстиля), хотя текст рядом обещал «ИИ
найдёт правильный код» для любого товара.

**Что сделано.** Скачан и подключён настоящий официальный международный
датасет Harmonized System (источник — `datasets/harmonized-system` на
GitHub, данные из UN Comtrade, открытая лицензия ODC PDDL) — 5613 кодов на
6-значном уровне, это основа кодов ТН ВЭД всех стран СНГ. Поиск теперь
реально работает по любому товару, присутствующему в этой номенклатуре —
практически по любому товару, который в принципе пересекает границу, а
не только по прежним шести.

**Двуязычность.** Официальные названия в датасете — на английском (язык
международного стандарта). Платформа не переводит самостоятельно почти
6000 наименований — машинный перевод специализированной таможенной
терминологии создаёт риск содержательной ошибки, которая может привести к
неверной классификации товара (это юридически значимо: неправильный код
ТН ВЭД означает неправильную таможенную пошлину). Вместо полного перевода
реализован словарь частых русских терминов (~150 товарных групп — зерно,
металлы, текстиль, нефтехимия, электроника, продовольствие, транспорт,
стройматериалы и т.д.), который переводит запрос в английские ключевые
слова для поиска по официальному датасету. Прямой поиск по коду и по
английскому названию работает без ограничений словаря.

**Честная граница, явно показанная в UI.** Платформа не утверждает, что
найденный код — единственно верный для таможенного оформления: показано
прямое предупреждение, что финальную классификацию по национальному
ТН ВЭД (с учётом дополнительных цифр конкретной страны) должен подтвердить
декларант или таможенный брокер. Убраны и вымышленные «стандартные
характеристики» товара (влажность, прочность и т.п.), которые раньше
автоматически подставлялись для старых 6 позиций, — настоящий датасет не
содержит технических спецификаций, только код и название, поэтому это
поле теперь честно оставляется пустым для заполнения пользователем вручную.

**Техническая защита размера платформы.** Сам датасет — почти 900KB в виде
файла. Подключён через отложенную (lazy) загрузку: грузится только когда
пользователь реально открывает вкладку поиска ТН ВЭД, не на каждой
загрузке страницы — иначе он удвоил бы размер первой загрузки платформы,
что прямо противоречило бы более ранней работе по сокращению этого
размера (см. CHANGELOG, 🔴#4). Подтверждено: основной чанк платформы —
666.98KB, датасет — отдельный чанк 859.93KB, не подгружается заранее.

**Найдена и исправлена проблема качества поиска в процессе работы.**
Первая версия поиска матчила переведённые слова как произвольную
подстроку — из-за этого короткие переводы давали ложные совпадения
(«tin» для «олово» совпадало внутри слова «whitings», «car» для
«автомобиль» совпадало внутри «carcasses» — мясо). Исправлено: поиск
теперь матчит по границе слова, с отдельным правилом для коротких (3-4
символа) переводов, требующих точного совпадения целого слова, чтобы не
ловить случайные совпадения внутри других слов.

Проверено: ключевые тестовые запросы (насос, олово, кофе, банан,
удобрения, прямой код 8471) дают релевantные результаты на первых
позициях. Известное ограничение: словарь покрывает частые товарные
группы, не все 5613 позиций — нишевые/неочевидные товары на русском могут
не находиться без ввода на английском или кода напрямую; это явно описано
пользователю в самом интерфейсе, не скрыто.

### Критическая регрессия найдена и исправлена в этой сессии

Основатель сообщил: маркетплейс не открывался — чёрный экран при нажатии,
платформа полностью перестаёт отвечать.

**Причина (моя ошибка в сессии 4, не новый баг в проекте):** при переводе
`canBuy`/`canSell` с устаревшего чтения `localStorage` на module scope на
реактивный хук `useAccountType()` (закрытие пункта 🟠#6), значения остались
доступны только внутри главного компонента `Marketplace()`. Два других
компонента в том же файле — `ProductCard` (карточка товара в списке) и
`ProductModal` (окно детального просмотра при клике на товар) — продолжали
напрямую обращаться к `canBuy`, которой в их области видимости больше не
существовало. До перевода на хук это работало случайно, потому что
переменная была глобальной для всего файла; после перевода — переменная
стала недоступна, и любой клик на товар (открывающий `ProductModal`) или
сам рендер списка товаров (использующий `canBuy` в `ProductCard`) вызывал
необработанную ошибку JavaScript, которая в отсутствие ErrorBoundary (см.
открытый пункт #13) обрушивала всё React-дерево страницы целиком — отсюда
чёрный экран и полная неработоспособность платформы.

Подтверждено прямым серверным рендером компонента в Node (минуя браузер) —
ошибка `canBuy is not defined` воспроизвелась точно в `ProductCard` на
строке использования. Исправлено: оба компонента (`ProductCard`,
`ProductModal`) теперь получают `canBuy` как явный prop от родителя, а не
обращаются к нему напрямую. Повторный тест рендера — для обоих типов
аккаунта (покупатель и продавец) — прошёл успешно.

**Урок для будущих сессий:** при переносе любой переменной с module/file
scope на React-хук внутри конкретного компонента нужно явно проверять, не
читают ли её другие функции-компоненты в том же файле напрямую (а не через
props) — такие связи легко упустить, потому что в JavaScript они работают
молча до первого реального использования сломанного пути, а не сразу при
сборке/компиляции.

Проверено сборкой: `npm run build` проходит, бандл не нарушен.

**Новый критический функционал — скрининг санкционных товаров (вне
исходного списка аудита, найдено и заказано основателем напрямую).**
Основатель лично протестировал платформу и обнаружил: ни при создании
тендера, ни при размещении товара в маркетплейсе не происходило никакой

true` в `marketplace.js` — статичные демо-данные, никак не связанные с
реальным пользовательским вводом; это уже было отражено как известное
ограничение в `BUSINESS_RULES.md` (строки 223, 294), но без реального
исправления. Без какой-либо защиты на старте реальной платформы это означает
прямой риск фасилитации санкционной торговли — то есть платформа сама
оказалась бы под санкциями за считаные часы после реального запуска.

Создан `src/utils/sanctionsScreening.js` — единый модуль `screenForSanctions()`,
используемый и в `Marketplace.jsx`, и в `CreateTender.jsx` (без дублирования
логики). Реализована честная двухуровневая модель, без претензии на то, что
keyword-matching заменяет настоящую экспортную классификацию:

- **HARD BLOCK** — однозначные категории (вооружение, военная техника,
  ядерные/химические/биологические материалы), запрещённые к гражданской
  B2B-торговле во всех основных режимах контроля без исключений. Публикация
  блокируется полностью (`disabled` на кнопке), без возможности продолжить
  через UI.
- **REVIEW REQUIRED** — категории двойного назначения (по структуре Wassenaar
  Arrangement / EU Regulation 2021/821 Annex I: микроэлектроника, телеком,
  навигация, отдельные химикаты, авиационные/аэрокосмические компоненты,
  металлы военной спецификации, нефтепродукты повышенного контроля).
  Публикация заблокирована до явного подтверждения пользователем чекбоксом
  («я проверил самостоятельно...») — платформа не утверждает, что товар
  «чист», только что он требует ручной проверки.

UI интегрирован в обе формы (`Marketplace.jsx` → `AddProductModal`,
`CreateTender.jsx` → шаг «Депозит» перед кнопкой публикации): явные блоки
предупреждения с понятным текстом на русском, кнопка публикации становится
`disabled` при блокировке.

**Найден и исправлен Rules of Hooks баг в процессе этой же работы**: при
первой версии правки `CreateTender.jsx` хук `useState` для
`confirmedReview` был по ошибке помещён ПОСЛЕ условного `return` (для
случая `!canBuy`) — это нарушение Rules of Hooks (разное количество хуков
